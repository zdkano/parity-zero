# Trust Model & Output Semantics

This document explains what parity-zero outputs mean, what is authoritative, and what is not.

## Key Concepts

parity-zero produces several distinct types of output. They have different trust levels and different purposes. Do not confuse them.

| Output Type | Authoritative? | Affects Scoring? | In ScanResult JSON? | In Markdown? |
|---|---|---|---|---|
| **Finding** | Yes | Yes | Yes | Yes |
| **ReviewConcern** | No | No | No | Yes |
| **ReviewObservation** | No | No | No | Yes |
| **Provider candidate notes** | No | No | No | Yes (when present) |
| **ReviewTrace** | No | No | No | No |

## Finding

A `Finding` is a concrete, actionable security issue detected in changed code.

- Findings are the **authoritative** output of the reviewer.
- Each finding has a `category`, `severity`, `confidence`, `file`, and `description`.
- Findings appear in the `ScanResult` JSON contract and in the markdown summary.
- Findings **drive scoring** — the `decision` (pass/warn/block) and `risk_score` are derived entirely from findings.
- Today, findings come only from **deterministic checks** (secrets detection, insecure configuration patterns). They are not generated by provider output.

### Categories

From the [findings taxonomy](../.squad/context/findings-taxonomy.md):

- `authentication` — missing or weak identity verification
- `authorization` — missing or weak access control
- `input_validation` — unsafe input handling
- `secrets` — exposed credentials or sensitive data
- `insecure_configuration` — unsafe defaults or settings
- `dependency_risk` — risky dependency usage

### What findings are NOT

Findings are not vague security prose. Each finding must be:
- tied to a specific file and (where possible) line range
- categorised according to the taxonomy
- rated for both severity and confidence independently

## ReviewConcern

A `ReviewConcern` is a **plan-level** contextual observation about an area that may deserve closer security attention.

- Concerns are **not authoritative** — they do not claim a proven issue exists.
- Concerns are **not findings** — they do not appear in `ScanResult` or affect scoring.
- Concerns appear in the markdown summary only.
- Concerns are derived from the `ReviewPlan` — they reflect signals from baseline context, sensitive paths, auth patterns, and review memory.

**Example:** "This PR modifies authentication-related code in a repository with known auth patterns. Consider reviewing session handling changes carefully."

## ReviewObservation

A `ReviewObservation` is a **per-file** contextual analysis note derived from `ReviewBundle` evidence.

- Observations are **not authoritative** — they describe why a file is security-relevant, not that it has a proven issue.
- Observations are **not findings** — they do not appear in `ScanResult` or affect scoring.
- Observations appear in the markdown summary only.
- Observations connect file-level review reasons, focus areas, baseline context, and memory context into concise explanations.
- When provider output is available, observations may be **enriched** with provider detail (marked as `+provider_enriched`). Enriched observations use hedged language.

**Example:** "src/auth/login.py is in a sensitive path (src/auth/) and the PR touches authentication logic in a Flask app with known JWT patterns."

## Provider Candidate Notes

When a reasoning provider (GitHub Models, Anthropic, or OpenAI) is enabled and invoked, it produces **candidate notes**.

- Candidate notes are **non-authoritative** — they are contextual material, not trusted findings.
- Candidate notes **do not create findings**, affect scoring, or influence the pass/warn decision.
- Candidate notes may appear in the markdown summary as additional observations.
- Candidate notes may enrich existing observations (via the observation refinement layer).
- The trust model is **identical across all providers** — no provider's output is treated as more authoritative than another's.
- Provider invocation is **gated** — the provider is only called when the PR context is rich or security-relevant enough to justify it.

### Why provider output is non-authoritative

Provider output comes from language models. It may be:
- insightful and useful context
- vague or generic
- incorrect

parity-zero treats provider output as candidate material that can inform human review. It never treats provider output as ground truth. This is a deliberate trust boundary — see ADR-025.

## Scoring

The `decision` and `risk_score` in `ScanResult` are derived **entirely from findings**.

- `pass` — no findings, or only low-severity findings
- `warn` — medium-severity findings present
- `block` — high-severity findings present

The scoring function is deterministic. It does not incorporate provider output, concerns, observations, or any other non-authoritative signal. This is an invariant enforced by the validation harness.

## ScanResult Contract

`ScanResult` is the structured JSON output of every review run. It contains:

- `scan_id` — unique scan identifier
- `repo` — repository identifier
- `pr_number` — pull request number
- `commit_sha` — head commit SHA
- `ref` — head branch ref
- `timestamp` — scan time (UTC)
- `decision` — pass / warn / block
- `risk_score` — 0–100
- `findings` — list of `Finding` objects

**What is NOT in ScanResult:**
- ReviewConcern — internal to markdown only
- ReviewObservation — internal to markdown only
- Provider candidate notes — internal to markdown only
- ReviewTrace — internal only, not exposed

## What Is Implemented vs Deferred

### Implemented
- Deterministic findings (secrets, insecure configuration)
- Structured ScanResult JSON contract
- Contextual concerns from review planning
- Per-file observations from review bundles
- Provider-backed observation enrichment
- Provider gating based on context richness
- ReviewTrace for internal traceability
- Validation scenario harness

### Intentionally Deferred
- Provider-generated findings (provider output does not create findings today)
- Provider output affecting scoring
- Confidence calibration for provider notes
- Full persistence (findings store, memory persistence)
- Control plane dashboard
- Policy-based review configuration

## Backend Persistence and Trust

The backend API persists review results (scan metadata and findings) for later retrieval. This persistence **does not change the trust semantics** of any output:

- Persisted findings remain findings — they have the same authority they had when produced
- Persisted concerns and observations are **not** stored (they are markdown-only)
- Storing a result does not make it more authoritative
- The backend is a **storage layer**, not an analysis or policy layer
- Retrieval of stored results reflects what was originally produced — no re-scoring or re-classification occurs

The backend stores the `ScanResult` JSON contract faithfully. It does not modify, filter, or enrich results.

### Run Summary Metadata (ADR-036)

In addition to the ScanResult fields, the backend persists **run summary metadata** per run:

- `provider_invoked` — whether the reasoning provider was called
- `provider_gate_decision` — gate outcome (invoked / skipped / disabled / unavailable)
- `concerns_count`, `observations_count` — how many non-authoritative items were generated
- `provider_notes_count`, `provider_notes_suppressed_count` — raw and filtered provider notes
- `changed_files_count`, `skipped_files_count` — file discovery and loading stats

These fields are **operational metadata for debugging and history**. They do not change the trust semantics of any output. Concern/observation/note text is not stored — only counts. Skipped-file paths and reasons are not stored — only the total count. This keeps the storage shape minimal while giving operators visibility into pipeline behaviour.
