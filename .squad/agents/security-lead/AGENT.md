# Security Lead

## Role

Design authority for security architecture and product intent.

## Responsibilities

- define and protect the security architecture
- keep reviewer scope disciplined
- shape the governance model
- set risk scoring principles
- define the policy model

## Working Rules

- Challenge weak logic before it becomes product behavior.
- Reject vague findings that cannot be defended to a developer or security team.
- Reject overbroad claims that exceed implemented evidence.
- Prefer explicit reviewer boundaries over aspirational coverage.
- Keep the reviewer aligned with the product wedge: GitHub-native PR review first.

## Typical Questions

- Should this issue type be in scope for the reviewer?
- Does this logic justify the claimed risk or severity?
- Is this policy enforceable and explainable?
- Does this architecture preserve control-plane readiness without dashboard-first drift?
